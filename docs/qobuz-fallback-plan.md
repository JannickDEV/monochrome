# Qobuz Fallback Plan

## 1. Architecture & Provider Abstraction
Monochrome currently lacks a formal multi-provider abstraction; `MusicAPI` (`js/music-api.js`) delegates directly to `LosslessAPI` (`js/api.js`), which hardcodes TIDAL API and Dev Mode `hifi-api` proxy calls. To introduce Qobuz as a first-class primary source in Dev Mode with TIDAL as a resilient fallback, we will establish a formal `Provider` interface (`js/services/types.ts`) defining core music operations (`search`, `getTrack`, `getAlbum`, `getArtist`, `getPlaylist`, `getStreamUrl`, `getTrackMetadata`). We will implement `TidalProvider` (`js/services/tidal/TidalProvider.ts`) to wrap existing Dev Mode hifi-api calls, `QobuzProvider` (`js/services/qobuz/QobuzProvider.ts`) to serve Qobuz catalog and streaming data, and `FallbackProvider` (`js/services/fallback/FallbackProvider.ts`) to execute operations sequentially across providers in priority order (Qobuz → TIDAL → Error) on a per-operation basis.

## 2. Dev Mode & Stream URL Resolution
In Dev Mode (`devModeSettings.isEnabled()`), REST requests route to an upstream proxy to bypass browser CORS preflight rejections and Cloudflare Turnstile challenges. Analysis of the user's `hifi-api` repository confirmed it is built on Python 3 / FastAPI / uvicorn / httpx, exposing routes with trailing slashes and query parameters (e.g., `/info/?id=...`, `/trackManifests/?id=...`, `/album/?id=...`). To ensure seamless multi-provider routing, our self-hosted `qobuz-api` service (Node/Express) is built to mirror these exact URL conventions (`/info/`, `/trackManifests/`, `/album/`, `/artist/`, `/playlist/`, `/search/` with query param `id`). Furthermore, we hardcode Dev Mode to be enabled by default in `js/storage.js`, pointing to the following default proxy endpoints:
- **Hifi-API**: `https://hifiapi.bitperfect.remotewire.net`
- **QobuzAPI**: `https://qobuzapi.bitperfect.remotewire.net`

We will wire `LosslessAPI` in Dev Mode to direct requests through `FallbackProvider([qobuzProvider, tidalProvider])`. For streaming, `getStreamUrl` will query Qobuz first—mapping Monochrome quality tiers to Qobuz format IDs (MP3 320 → 6, FLAC 16/44.1 → 7, FLAC 24/≤96 → 8, FLAC 24/≤192 → 13, FLAC 24/>192 → 27)—and fall back to TIDAL only when Qobuz returns an unusable result (404, network error, or missing stream URL).

## 3. Settings UI & Observable Fallback
We will extend `devModeSettings` (`js/storage.js`) and the settings interface (`js/settings.js`) to store Qobuz credentials (`appId`, `userAuthToken`, and local service URL `qobuzUrl`, defaulting to `https://qobuzapi.bitperfect.remotewire.net`), including an interactive "Test connection" button that validates credentials against `/user/get`. To ensure observability and debuggability, every fallback transition from Qobuz to TIDAL will be explicitly logged with operation name, error class, and status. Additionally, we will introduce a dynamic source indicator ("Source: Qobuz / TIDAL") in the player and search UI to visually reflect which provider served the active item or audio stream.

## 4. Testing & Verification Strategy
The repository uses Vitest for unit testing (`js/tests/`). We will add unit tests for `QobuzClient` (verifying required headers, concurrency queuing, and 429 exponential backoff), `QobuzProvider`, and `FallbackProvider` (verifying that Qobuz success avoids TIDAL calls, Qobuz errors trigger TIDAL fallback, and dual failures throw typed `ProviderError`s). We will verify that when Qobuz credentials are absent, the application behaves exactly as today (TIDAL only), and ensure all linting, typechecking, and builds pass cleanly.
